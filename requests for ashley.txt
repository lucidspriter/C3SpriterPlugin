## SDK2 runtime access to custom plugin animation frames / atlas image info
- We can set `SetHasAnimations(true)` on the Spriter plugin object type and load atlas PNGs into the plugin's animation frames at import time (editor-side), but at runtime (SDK2) we have not found a documented API on `ISDKWorldInstanceBase` / `ISDKObjectTypeBase` / runtime `IObjectType` to enumerate that plugin object's animation frames (equivalent to legacy `GetObjectClass().GetAnimations()[0].GetFrames()`).
- Current runtime logs show legacy/internal-style access (`GetObjectClass/getObjectClass`) is unavailable in some environments, so self-draw must fall back to loading atlas PNGs from Project Files and slicing via SCON metadata.
- Question: Is there a supported SDK2 runtime API to access a custom plugin's animation frames/image infos for `SetHasAnimations(true)` object types, or is project-file loading the intended approach?
- References:
  - ISDKObjectTypeBase docs (shows `getImageInfo()` for `SetHasImage(true)`, but no documented animation-frame enumeration): https://www.construct.net/en/make-games/manuals/construct-3/scripting/scripting-reference/addon-sdk-interfaces/isdktypebase
  - ISDKWorldInstanceBase docs (no documented `GetObjectClass/getObjectClass`): https://www.construct.net/en/make-games/manuals/construct-3/scripting/scripting-reference/addon-sdk-interfaces/isdkinstancebase
  - Runtime `IObjectType` script interface docs (no animation list methods documented): https://www.construct.net/en/make-games/manuals/construct-3/scripting/scripting-reference/object-interfaces/iobjecttype
  - Editor Addon SDK `IObjectType` interface does expose `GetAnimations()` (editor-side only, not runtime): https://www.construct.net/en/make-games/manuals/addon-sdk/reference/objecttype

## ISDKWorldInstanceBase direct property setter parity vs. resolved world-info path
- During SDK2 cleanup, direct self-instance writes like `this.opacity = ...`, `this.isVisible = ...`, and `this.zElevation = ...` looked like the intended path from the scripting docs, but runtime validation in the Spriter addon regressed compared to routing through the resolved world-info/self-instance compatibility path.
- The docs now look more like two different official contexts:
  - Addon SDK runtime reference says runtime APIs are the same as Construct's scripting APIs
  - Addon SDK editor/reference pages expose method-style interfaces such as `GetX()`, `SetOpacity()`, `SetOriginX()`
- The remaining uncertainty is not "which doc family should runtime use" (the runtime reference answers that); it is why some documented runtime-style direct property writes did not behave as expected in this addon runtime context.
- Question: for Addon SDK v2 runtime instances, should direct `IWorldInstance` property setters always update the underlying engine state the same way as the resolved world-info path, or are there cases where addon runtime code should prefer another official setter route?

## SDK2 runtime APIs for instance enumeration, naming, UID, picking, siblings

The Spriter runtime needs to interact with associated Sprite object types at runtime for non-self-draw mode. Several operations require APIs that are not clearly documented for SDK v2 addon runtime code:

- **Instance enumeration**: Given an object type, enumerate its instances. Currently probing `instances()`, `getAllInstances()`, `GetInstances()`.
- **Object type name**: Get the name of an object type. Currently probing `GetName()` and the `.name` property.
- **Instance UID**: Get the UID of an instance. Currently probing `GetUID()`, `getUID()`, `getUid()`, and `.uid` property.
- **Picking/resolved instances**: Get the currently picked or paired instance of an object type in an event context. Currently probing `getPickedInstances()`, `getFirstPickedInstance()`, `getPairedInstance()`, `getFirstPicked()`.
- **Container siblings**: Get sibling instances from a container. Currently probing `GetSiblings()`, `getSiblings()`.
- **Sprite frame index**: Read the current animation frame index from a Sprite SDK instance. Currently probing `_changeAnimFrameIndex`, `_currentFrameIndex`, `currentFrameIndex` and methods like `_DoChangeAnimFrame()`.

Question: What are the supported SDK v2 runtime APIs for these operations when an addon needs to interact with other object types (particularly built-in Sprite) at runtime?
